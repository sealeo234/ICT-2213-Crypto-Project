# **ICT-2213-Crypto-Project**

Flask Python Web Server - VaulnZero

Implements a file sharing system where the server has no knowledge of the end user private keys or the plaintext of uploaded files.

The server maintains a table of encrypted files, and maintains an Access Control List (ACL) for file ownership and permissions.

The server also maintains a table of registered users, their password hashes and their UUIDs.

## Quick Setup

1. Optional: Create and activate a virtual environment:
   - Windows (PowerShell):
     - `python -m venv .venv`
     - `.\.venv\Scripts\Activate.ps1`
   - Linux / macOS:
     - `python3 -m venv .venv`
     - `source .venv/bin/activate`
2. Install dependencies:
   - `pip install -r requirements.txt`
3. Run the application:
   - `python app.py`
4. Open in browser:
   - Local: `https://127.0.0.1`
   - From another machine: Find the server's IP address and use `https://<server_ip>`
     - On Linux: `ip addr show` (look for `eth0` or similar interface)
     - On Windows: `ipconfig` (look for the network adapter IP)
     - Accept the self-signed certificate warning in your browser

## Technicals

**File Upload:**

* A key pair is generated by the client on registration. 
* Private key is stored in the IndexedDB variable within the memory space of the browser, and the user will also be prompted to save the private key as a .pem file. A passphrase is required to secure the private key.
* Public key is transmitted to the server via API call and stored in the server database.
* To upload a file, the client (owner) generates a **File Encryption Key (FEK)**, which is a **symmetric AES-GCM** key used to encrypt the file content.
* The file is encrypted locally in the browser using the FEK and a random **initialization vector (IV)**
* The client also signs the encrypted payload (`IV || ciphertext`) using the client's own **ECDSA P-256 signing private key**.
* The FEK is then encrypted (wrapped) with the public key of the owner. This produces a single wrapped key.
* Both the encrypted file and the wrapped key are transmitted to the server for storage. The server stores the encrypted file, its IV, and the wrapped key.
* Signature metadata (`signature`, `signature_alg`, `signer_public_key`) is uploaded and stored with each file.

**Permissions Modification:**
* The file owner may modify access to the file, the FEK is wrapped for the new set of recipients with each of their public keys. This produces a **set of wrapped keys**, one for each intended recipient, that is stored in the server.

**File Download:**
* When a recipient wants to download a file, the server only transmits the encrypted file, IV, and the wrapped AES key corresponding to that recipient.
* Before decryption, the client verifies the signature over (`IV || ciphertext`) using the stored signer public key.
* The recipient retrieves their private key from their browser’s IndexedDB (or imports it from a saved .pem file) to decrypt the FEK.
* Using the decrypted FEK and IV, the recipient decrypts the file locally in the browser, ensuring end-to-end encryption without the server ever seeing the plaintext content.

## Frontend JavaScript Structure

The frontend code is split by feature flow, while low-level cryptographic primitives are grouped under a dedicated `crypto/` subfolder.

**Feature Flow Files (`static/js/`):**

* `register.js` — registration flow and initial keypair generation/export trigger.
* `upload.js` — file upload orchestration, FEK-based encryption flow, wrapping, and signature metadata submission.
* `download.js` — file download orchestration, signature verification, FEK unwrapping, and decryption flow.
* `access-update.js` — owner access-list updates and FEK rewrapping for recipients.
* `key-rotate.js` — user key rotation and per-file rewrap process.
* `key-mgmt.js` — shared helpers (UI prompts/alerts, IndexedDB identity storage/load, encoding utilities, identity import/export).

**Low-Level Crypto Files (`static/js/crypto/`):**

* `fek.js` — AES-GCM and FEK-level operations.
* `key-wrapping.js` — RSA-OAEP key import/wrap/unwrap operations.
* `ecdsa-sign.js` — ECDSA key import/sign/verify operations.

## Security Properties

This design helps protect against:

* **Ciphertext tampering at rest/in transit**: modified encrypted bytes or IV fail signature verification.
* **Accidental corruption**: recipients detect corruption before decrypting.
* **Basic signer impersonation by normal users at upload**: backend requires uploaded signer public key to match the current authenticated user's registered signing key.
* **Payload substitution without signing key**: attackers who do not control the signer's private key cannot forge a valid signature over modified payloads.

## Vulnerabilities

This design does **not** fully protect against:

* **Strong malicious-server key-substitution scenarios**: verification currently trusts signer key metadata provided by the server for that file.
* **Server-side cryptographic enforcement on upload**: backend checks key identity fields but does not independently verify signature correctness before storing.
* **Client compromise (XSS/malware/browser compromise)**: if client private keys are stolen or JS execution is compromised, signatures and encryption guarantees can be bypassed.
* **Account/session takeover**: if an attacker controls the authenticated account/session, they can upload legitimately signed malicious content as that user.
* **General web threats outside crypto controls**: CSRF/XSS/access-control bugs still require separate web security hardening.
